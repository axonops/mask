# Corpus

Bulk fixture corpus for `github.com/axonops/mask`. One file per registered rule; each file pairs an input with the expected `mask.Apply` output, one fixture per line.

## Running

```sh
make test-corpus      # runs the harness (build tag: corpus)
make corpus-regen     # regenerates the generated section + .corpus.lock
```

`go test ./...` does NOT run the corpus — the build tag keeps it out of the regular test cycle. The full quality gate (`make check`) does run it.

## File layout

```text
tests/corpus/
├── corpus_test.go           # the harness (build tag: corpus)
├── .corpus.lock             # SHA-256 manifest, verified at TestMain
├── <rule>.txt × 68          # one fixture file per registered rule
├── gen/                     # generator (build tag: corpusgen)
│   ├── main.go              # entrypoint; preserves canonical, rewrites generated
│   ├── registry.go          # Generator interface + register() helper
│   └── <rule>.go × N        # per-rule input strategies
└── _tools/                  # author helpers (build tag: corpushelper)
    ├── bootstrap/           # one-off canonical scaffolder
    └── expect/              # stdin → input<TAB>expected helper
```

## Fixture file format

Each `<rule>.txt` has two sections divided by a banner:

```text
# === canonical (hand-written; preserved by generator) ===
#
# Rule: phone_number
# … description and notes …

# A handful of spec-pinned examples
+44 7911 123456    +44 **** **3456
0044 7911 123456   0044 **** **3456

# === generated (do not edit; regenerated by make corpus-regen) ===
#
# 250+ generated fixtures for phone_number ...
+1-555-...-1234    +1-***-***-1234
…
```

Hand-written content above the `# === generated ===` banner is preserved verbatim across `make corpus-regen`. Everything below it is rewritten from the generator.

Format rules (enforced by `TestCorpusFormatStrict`):

- UTF-8, LF line endings, no BOM.
- Exactly one TAB per fixture line.
- Comments start with `#`. Blanks ignored.
- Pragma `# corpus: escaped` (optional) enables `\\`, `\t`, `\n`, `\r`, `\xNN`, `\uXXXX` decoding for the rest of the file.

## Adding fixtures

### Hand-curate canonical inputs

```sh
printf 'one input per line\nfoo@bar.com\n07700 900123\n' \
  | go run -tags corpushelper ./tests/corpus/_tools/expect <rule>
```

The helper streams `input<TAB>mask.Apply(rule, input)` to stdout. Append the relevant block to the canonical section of `tests/corpus/<rule>.txt`.

### Add or extend a generator

```go
//go:build corpusgen

package main

type myRuleGen struct{}

func (myRuleGen) Generate() []Pair {
    r := rand.New(rand.NewPCG(0xC0FFEE, seedMyRule))
    var inputs []string
    // …deterministic input strategy…
    return uniqueLinesToPairs(inputs)
}

const seedMyRule uint64 = 0xDEC0DEXX  // unique constant

func init() { register("my_rule", myRuleGen{}) }
```

Generators MUST be deterministic — fixed PCG seed, sorted map iteration, no `time.Now()`. The `corpus-lock-fresh` CI job will fail if a re-run produces a diff against the committed `.corpus.lock`.

## Latent-bug discovery

If a generated fixture line looks suspicious (the rule under-redacts, leaks a separator, or fails closed on a valid input), DO NOT fix the generated line by hand. Instead:

1. Add a `# BUG?` comment above a representative input in the **canonical** section.
2. Open a follow-up issue describing the discrepancy.
3. Fix the rule in a separate PR — the corpus will reflect the fix at the next `make corpus-regen`.

The corpus codifies *current* behaviour by design. Hand-edits to the generated section are clobbered by the next regeneration.
